Voici un plan de route détaillé, pragmatique et ordonné pour mener ton projet webserv (C++98) de zéro à la version livrable. Je suis concis et factuel pour que tu puisses suivre étape par étape.

1) Lecture / préparation (1–3 jours)
- Lire les RFCs essentiels : RFC7230..7235 (HTTP/1.1), CGI spec, multipart/form-data, chunked transfer.  
- Tester NGINX et client basiques : curl, telnet, netcat. Comprendre les headers et comportements.  
- Installer outils : g++ (avec -std=c++98), valgrind, strace, gdb, make, wrk/ab (stress).  

2) Définir l’architecture et les contraintes (0.5–1 jour)
- Éviter tout blocage : un seul poll()/select()/epoll()/kqueue() pour tous les IO non‑bloquants.  
- Choisir l’API d’événements (poll pour portabilité, epoll pour Linux performant). Décision: implémenter poll() d’abord, puis adapter/optimiser pour epoll/kqueue si besoin.  
- Respecter C++98, flags -Wall -Wextra -Werror, pas de libs externes.

3) Organisation du projet (fichiers / dossier)
- srcs/, includes/, configs/, www/ (site de test), tests/, bin/  
- Fichiers minimaux : Makefile, main.cpp, Server.hpp/.cpp, ConfigParser.hpp/.cpp, Connection.hpp/.cpp, HttpRequest.hpp/.cpp, HttpResponse.hpp/.cpp, Router/Handler modules, CGI module, Utils.  
- Exemples de configs dans configs/ (default.conf, tests.conf).

4) Phase 0 — bootstrap et Makefile (0.5 jour)
- Créer Makefile minimal (all, clean, fclean, re, name).  
- Compiler un binaire vide ./webserv qui accepte 0 ou 1 argument (config).  
- Ajouter options de compilation strictes (-std=c++98 -Wall -Wextra -Werror).  

5) Parser de configuration (2–4 jours)
- Parser simple inspiré d’un bloc server/nginx (listen, root, index, error_page, client_max_body_size, location blocks avec allowed_methods, autoindex, upload_path, cgi ext, redirect).  
- Représenter la config en structures C++ (servers vector, each with vector<listen_endpoints>, locations).  
- Ajouter validations et messages d’erreur lisibles.  
- Fournir configs de test qui couvrent cas demandés.

6) Conception du cœur réseau — accept et boucle d’évènements (3–6 jours)
- Fonctions à implémenter :
  - Création et binding de sockets pour toutes les interface:port de config.
  - set non‑blocking (fcntl O_NONBLOCK), setsockopt SO_REUSEADDR, close on exec si utile.
  - Utiliser poll()/epoll() pour surveiller tous les sockets (inclut sockets d’écoute).
  - Accept non‑bloquant, créer Connection objects par client.
  - Maintenir map<fd, Connection*>.
- Gestion des évènements : lecture seule si readable, écriture si writable; jamais read/write sans poll() l’ayant signalé.
- Gérer EAGAIN/EWOULDBLOCK, ECONNRESET, fermeture propre des clients.

7) Module Connection & buffers (2–4 jours)
- Socket non‑bloquant, deux buffers : read_buffer, write_buffer.  
- Lecture incrémentale : parser incrémental de requête HTTP (request line + headers, body selon Content-Length / chunked).  
- Écrire via send() en boucle jusqu’à ce que tout soit envoyé; gérer écritures partielles.

8) Parser HTTP robuste (3–6 jours)
- Parser d’état (start-line, headers, body). Support minimal : GET, POST, DELETE.  
- Gérer Content-Length, Transfer-Encoding: chunked (déchunker côté serveur).  
- Ne jamais consulter errno après read/write pour modifier logique (sujet interdit). Gérer en se basant sur valeurs de retour et comportements documentés.

9) Routage & handlers (3–6 jours)
- Pour une requête : trouver matching server (port) et location (prefix matching), appliquer règles (allowed methods, redirect).  
- Handlers :
  - Static file: vérifier access(), stat(), ouvrir/read et renvoyer (fichier sur disque = lecture bloquante permise). Gérer index files et autoindex (genère HTML listing).
  - Uploads (POST): sauvegarder dans upload_path, vérifier max body size.
  - DELETE: supprimer fichier (access/stat/unlink).
  - CGI: spawn process via fork() + execve() (autorisé uniquement pour CGI). Créer pipes non‑bloquants, passer variables d’environnement, body à stdin, récupérer stdout. Attention aux descriptors et close-on-exec.
- Respecter codes de statut HTTP corrects (200, 201, 204, 301/302, 400, 404, 405, 413, 500, 502, 504…).

10) CGI details (2–4 jours)
- Préparer envp (REQUEST_METHOD, CONTENT_LENGTH, SCRIPT_NAME, QUERY_STRING, PATH_INFO, SERVER_PROTOCOL, CONTENT_TYPE, etc.).  
- fork() puis execve() le binaire CGI. Use pipes pour stdin/stdout. Fermer/dupliquer descripteurs (dup2).  
- Non‑bloquant sur pipes et surveiller via poll() en même boucle évènements (la règle “un seul poll()” s’applique aussi aux pipes).  
- Gérer fins/EOF correctement (CGI peut signaler fin via EOF si pas Content-Length).  
- Timeout pour CGI afin d’éviter blocage infini.

11) Erreurs, pages d’erreur et limites (1–2 jours)
- Pages d’erreur par défaut si non fournies.  
- Respecter client_max_body_size (renvoyer 413).  
- Timeout de requêtes et connexions inactives.

12) Tests unitaires & d’intégration (continu)
- Tests unitaires simples pour parser config et parser HTTP (en C++ ou petits programmes de test).  
- Tests manuels avec curl/telnet pour: GET files, directory listing, POST upload multipart and raw, DELETE, CGI returning headers+body, chunked requests.  
- Scripts d’intégration pour automatiser tests.

13) Stress / robustesse / fuites (2–4 jours)
- Utiliser wrk/ab pour charge; valgrind et addr/asan (si possible) pour fuites.  
- Gérer limites des FDs, files d’attente, épuisement mémoire (ne jamais crash).  
- Timeout/garde-fou pour clients lents (slowloris).

14) Logging, signals et arrêt propre (1–2 jours)
- SIGINT/SIGTERM handler sets flag to stop; fermer listeners, noter connexions en cours, terminer proprement.  
- Log minimal (stderr/stdout) : errors, accepts, closes, request summary.

15) Polishing & conformité (1–3 jours)
- Vérifier compilation C++98 (éviter features C++11+).  
- Revue code: éviter recursion profonde, commentaires, const correctness, RAII pour FDs (destructors closes).  
- Makefile ne relie pas inutilement, variables, V=1 option.

16) Livraison & dossier (0.5–1 jour)
- Fournir configs de test, site statique complet, scripts de test, README expliquant usage, endpoints, limitations connues.  
- S’assurer que ./webserv [config] fonctionne.  
- Packager (fclean + make) et vérifier sur machine propre.

Checklist finale avant rendu
- [ ] Ne pas crash en cas d’erreur ou ressource low.  
- [ ] Un seul mécanisme d’évènements non‑bloquant pour sockets/pipes (poll/epoll/kqueue).  
- [ ] Respect des méthodes GET/POST/DELETE, uploads, CGI support.  
- [ ] Pages d’erreur par défaut et status corrects.  
- [ ] Multi‑ports supportés (plusieurs listen endpoints).  
- [ ] Tests automatisés et stress passés localement.  
- [ ] Makefile complet + -std=c++98 compile clean.

Conseils techniques clés (rappels)
- Toujours set O_NONBLOCK sur sockets et pipes contrôlés par poll.  
- Ne jamais read/write sur sockets sans poll.  
- Ne pas se baser sur errno pour read/write après poll (subject interdit); base-toi sur valeur retour et logique de l’état.  
- Utilise buffers pour reads/writes et gère cas partiels.  
- Utilise RAII pour FDs (classe wrapper) afin d’éviter leaks dans exceptions.  
- Documente comportements non‑implémentés et limites.
